### Решение 1

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

Для бюджетирования проектов и формирования финансовых отчётов подойдут реляционные СУБД (например, PostgreSQL или Microsoft SQL Server), так как они обеспечивают высокую целостность данных и поддержку транзакций, что критично для финансовых данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 

Для ускорения работы с хешированием можно использовать API для параллельной обработки данных, такие как Apache Hadoop или Apache Spark.

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

Для лендингов и CRM лучше всего использовать гибкие и быстрые NoSQL СУБД, например MongoDB или Cassandra. Они позволят эффективно обрабатывать большие объёмы данных и быстро осуществлять операции по добавлению и извлечению информации.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

Да, эту задачу можно решить одной СУБД. Например, можно использовать MongoDB для хранения данных по лендингам и CRM, так как она поддерживает гибкую структуру данных и обладает высокой производительностью.

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

Для отдела контроля качества подойдёт реляционная СУБД, например PostgreSQL или MySQL, с простой и понятной структурой. Они позволят удобно хранить и организовывать базу по корпоративным нормам и правилам, обучающему материалу и другой информации.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

Да, можно использовать уже существующую реляционную СУБД из предыдущих задач, если в ней создать отдельные таблицы для хранения данных отдела контроля качества с учётом их структуры.

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

Для решения задач логистики, связанных с формированием маршрутов доставки и распределением курьеров, лучше всего подойдут графовые СУБД, например Neo4j. Они специализируются на работе со связями между данными и позволяют эффективно моделировать и анализировать маршруты.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

Можно подключить отдел закупок к графовой СУБД логистики, если данные отделов имеют общие связи. В противном случае, для отдела закупок лучше создать отдельную СУБД, которая будет оптимизирована под их специфические задачи.

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Да, все перечисленные задачи можно решить, используя одну мощную реляционную СУБД, такую как Oracle или Microsoft SQL Server. Она обладает широким спектром функциональности и возможностей для работы с различными типами данных.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

*Приведите ответ в свободной форме.*

### Решение 2

Процесс пополнения баланса счёта телефона может включать следующие шаги:

1. Аутентификация пользователя: Пользователь должен войти в систему платежного сервиса или мобильное приложение, используя свои учетные данные.
2. Ввод суммы пополнения: Пользователь вводит желаемую сумму пополнения баланса.
3. Выбор метода оплаты: Пользователь выбирает предпочтительный способ оплаты (например, банковская карта, электронный кошелек, прямой банковский перевод и т.д.).
4. Подтверждение платежа: Пользователь проверяет введенные данные и подтверждает транзакцию.
5. Обработка платежа: Платежная система обрабатывает транзакцию и списывает средства с выбранного метода оплаты.
6. Подтверждение пополнения: Мобильный оператор получает информацию о платеже и пополняет баланс телефона, после чего отправляет пользователю уведомление о успешном пополнении.

### Решение 3

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

Преимущества SQL-систем по отношению к NoSQL:

1) Структурированный язык запросов: SQL предлагает удобный и понятный язык запросов для работы с базами данных, что делает его более простым для использования и понимания.
2) Надежность и целостность данных: SQL-системы обеспечивают ACID-свойства (атомарность, согласованность, изолированность, долговечность), гарантирующие надежность и целостность данных.
3) Поддержка транзакций: SQL-системы обеспечивают механизм транзакций для выполнения нескольких операций как единое целое, что помогает избежать некорректных изменений данных.
4) Широкий выбор инструментов и библиотек: SQL-системы имеют развитую экосистему инструментов и библиотек для работы с данными, что упрощает разработку и управление базами данных.
5) Поддержка сложных запросов: SQL позволяет выполнять сложные запросы к данным, включая соединения таблиц, группировку, агрегирование, фильтрацию и сортировку.

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

*Приведите ответ в свободной форме.*

### Решение 4

При выборе типа СУБД для обработки огромного количества данных и выполнения большого количества вычислений на 1000 машинах следует учитывать следующие критерии:

1) Масштабируемость: СУБД должна обеспечивать возможность горизонтального масштабирования, чтобы эффективно работать с таким объемом данных и вычислений.

2) Производительность: СУБД должна обеспечивать высокую производительность при выполнении запросов и обработке данных.

3) Надежность: СУБД должна быть надежной и обеспечивать сохранность данных в случае сбоев или отказов.

4) Гибкость: СУБД должна поддерживать различные модели данных и типы запросов для работы с разнообразными данными.

Исходя из этих критериев, можно рассмотреть использование распределенных вычислений на базе Apache Hadoop или Apache Spark. Обе эти технологии обладают хорошей масштабируемостью, производительностью и надежностью при обработке больших объемов данных.

Apache Hadoop предлагает фреймворк MapReduce для параллельной обработки данных на кластерах серверов, что позволяет эффективно использовать вычислительные ресурсы 1000 машин.

Apache Spark, в свою очередь, предлагает более высокую производительность благодаря встроенному кэшированию данных в оперативной памяти и возможность работать с различными типами данных (структурированными, полуструктурированными и неструктурированными).

Таким образом, использование распределенных вычислений на базе Apache Hadoop или Apache Spark на 1000 машинах будет наилучшим выбором для обработки большого объема данных и выполнения вычислений.
